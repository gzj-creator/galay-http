### WsEcho Fewer Wakeups 复跑快照

随后又在 `galay-http/kernel/websocket/WsConn.h` 里加入了更窄的 composite echo path，用一次内部 `ReadWrite` state machine 覆盖服务端常见的 `getMessage() -> sendText()/sendBinary()` 往返，并补了 `T62-ws_echo_fewer_wakeups` 与 `T47/T55/T60/T61/T62` fresh 回归。该实现中一度存在 `WsReaderSetting` 生命周期错误，导致 plain `ws` benchmark 下出现 `Message too large` 和大量失败；在把 reader state 改为按值持有 setting 后，重新执行同一组 `ws` / `wss` Galay vs Rust fresh 对比。

| 协议 | Galay | Rust | Galay 相对 Rust | 备注 |
| --- | ---: | ---: | ---: | --- |
| `ws` | `97399.88 rps` | `136268.50 rps` | `-28.52%` | 相比用户指定的有效基线 `ws-writer-reuse` 的 `96164.50 rps` 小幅提升到 `97399.88 rps`，绝对吞吐 `+1235.38 rps`；同时保持 `fail=0`，证明 composite path 的生命周期问题已修正 |

这一轮同样是 `fail=0`。最新 fresh 结果显示 `fewer wakeups` 这一层在当前机器和这次 clean rebuild 后带来了小幅绝对吞吐提升，但提升量并不大，仍不足以改变 plain `ws` 明显落后 Rust 的事实。更稳妥的结论是：WebSocket 层的组合 awaitable 现在已经能正确工作，并且可能回收了一部分固定成本；但如果要继续缩小和 Rust 的差距，下一步仍应优先排查 runtime / scheduler / reactor 侧的固定开销，而不是只在当前 `WsConn` 组合路径上继续做库层微调。
